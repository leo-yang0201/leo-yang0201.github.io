Http

Http常见面试题

Http与Https的区别

  • http的url以http://开头，https以https://开头
  • http是不安全的（明文传输），https安全
  • http标准端口号80，https 443
  • osi模型里，http工作与应用层，https的安全传输工作在传输层
  • http无法加密，https对传输数据进行加密
  • http无需证书，https需要CA机构颁发的SSL数字证书

常用的HTTP方法有哪些？

  • GET： 用于请求访问已经被URI识别的资源，通过URL传参给服务器，get参数通过url传递，传递的参数有长度限制，请求参数会被保留在历史记录里，一次产生一个tcp数据包，对于get请求，浏览器先将http header和data一起发送，服务器响应200 ok，用户获取数据不用每次都连接数据库，可以用缓存
  • POST：向指定资源提交数据，请求服务器去处理，post参数放在request body里，没有长度限制，请求可以有多种编码方式，参数不会保留在历史记录，一次产生两个tcp数据包，对于post请求浏览器先发送header，服务器响应100 continue，浏览器在发送data，服务器响应200 ok，post一般用于更新删除操作，不能用缓存
  • HEAD：获取服务器的响应头信息，常用于查看服务器的性能
  • OPTIONS：请求服务器返回该资源所支持的所有HTTP请求方法，常用于客户端查看服务器的性能
  • PUT：传输文件，报文主体中包含文件内容，保存到对应URI位置
  • DELETE：删除文件，与PUT方法相反，删除对应URI位置的文件

HTTP2.0新特点？

  • 基于https，天然安全
  • 二进制分帧层，将所有传输信息分割成更小的信息或帧，并进行二进制编码（增强性能）
  • 允许多路复用，基于二进制分帧层，在共享tcp连接基础上，同时发送请求和响应。http消息被分解为独立的帧，不破坏本身的语义，交错发送出去，最后在另一端根据流ID和首部将他们重新组合起来
  • 服务器推送：服务端根据客户端的请求，提前返回多个响应，推送额外的资源给客户端，支持缓存（遵循同源策略；基于客户端的请求响应来确定的）

HTTP缓存机制？

  • 强缓存（200）：浏览器不向客户端发送任何请求，直接从本地缓存中读取文件并返回（相关字段：Cache-Control，Expires）
  • 协商缓存（304）：浏览器发送请求到服务器，通过服务器告知缓存是不是可用（相关字段：Last-Modified/If-Modified-Since，Etag/If-None-Match）

HTTP无状态协议

http没有记忆能力，缺少状态意味着如果处理后续的信息需要签名的信息

当客户端一次http请求完成以后，客户端再发送一次http请求，http并不知道当前客户端是一个老用户

可以使用cookie解决无状态问题

Cookie和Session的共同点和区别？

  • Cookie保存在客户端（浏览器），Session数据保存在服务器
  • Cookie没有Session安全，本地的cookie存在被攻击的风险，考虑安全使用Session
  • Session会一段时间内保存在服务器上，当访问增多则占用服务器性能，考虑性能使用cookie

CSRF和XSS攻击手段

  • CSRF（跨站请求伪造）：攻击者在页面里嵌入恶意js脚本，当用户浏览页面时进行攻击。解决方案：cookie设置httponly和secure字段，进行特殊字符过滤，对用户的输入进行检查
  • XSS（跨站脚本攻击）：攻击者盗用用户身份，以用户名义发送恶意请求。解决方案：使用验证码，检查https头部referer字段

Cookie如何防范XSS攻击？

在http头部配上set-cookie，其中httponly属性会禁止js脚本使用document.cookie访问cookie

secure属性告诉浏览器仅在请求为https时发送cookie

在地址栏里输入一个URL,到这个页面呈现出来，中间会发生什么？

简答：DNS解析 -> TCP连接 -> 发送http请求 -> 服务器处理请求并返回http报文 -> 浏览器解析渲染页面 -> 连接结束

详细：

  • 根据URL域名寻找服务器ip，浏览器现在浏览器缓存里找，再去系统缓存找，再去路由器缓存找，都找不到再去系统的hosts文件中查找，还没有就查DNS服务器
  • 得到ip地址后，浏览器根据ip和端口后构建一个http请求并将该请求封装在一个tcp包里，这个tcp包依次经过传输层、网络层、数据链路层、物理层到达服务器，服务器解析这个请求并做出响应，返回对应的html给浏览器
  • 浏览器根据返回的html构建DOM树，并行下载音频视频图片资源，遇到js脚本则会下载对应脚本，根据外部样式、内部样式、内联样式构建CSSOM树，和DOM树合并成渲染树
  • 进行布局，确定各元素位置和尺寸，渲染页面返回

HTTPS的工作原理

  • 客户端通过URL发起https请求，要求服务器建立ssl连接
  • 服务器收到客户端的请求，返回CA证书
  • 客户端检测公钥证书是否有效，如果无效发出警报，有效就利用伪随机数生成会话密钥，用证书的公钥加密会话密钥，发给服务器
  • 服务器用证书的私钥解密会话密钥，至此服务器和浏览器都有相同的会话密钥
  • 两端通过AES对称加密传递数据

HTTPS加密方式

使用非对称加密传输一个对称密钥，服务器和客户端用这个对称密钥来加密解密收发数据，而具体的传输数据通过对称加密方式

  • 对称加密AES：加密和解密用同一个密钥（速度快）
  • 非对称加密RSA：发送端使用公开的公钥加密，接收端使用私密的私钥解密（安全）

HTTP状态码

200 OK 请求成功

301 永久重定向 请求的信息已经被移动到新的URI，返回新的URI

302 临时移动 客户端继续使用原URI

304 未修改，服务器返回此状态码不会返回任何资源，所请求的资源未修改

400 bad request 客户端请求语法有误

401 当前请求需要用户验证

403 服务器收到请求但拒绝执行

404 服务器无法根据用户的请求找到资源

500 服务器内部错误
