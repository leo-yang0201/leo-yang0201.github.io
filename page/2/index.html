请你简述TCP、UDP的区别

TCP和UDP是OSI模型中的运输层中的协议。TCP提供可靠的通信传输，而UDP常被用于让广播和细节控制交给应用的通信传输

  • TCP面向连接，UDP面向非连接，即发送数据前不需要建立连接
  • TCP提供可靠的服务（数据传输无差错，不丢失，不重复，按序到达），UDP无法保证可靠交付
  • TCP面向字节流，UDP面向报文（对实时应用有用如实时视频会议）
  • TCP数据传输慢，UDP数据传输快
  • TCP连接只能是点到点的，UDP支持一对一，一对多，多对多，多对一的交互通信

TCP三次握手

举个例子说明，张三（客户端）首先对李四（服务器）招手（发送一个syn），李四看到张三向自己招手后，点头并挤出一个微笑（发送一个ack），张三看到李四微笑后确认李四成功辨认了自己（张三进入了establish状态），但李四不知道张三是不是向他招手，所以他也向张三招了手（发送一个syn），张三看到后知道李四在寻求自己的确认，于是也点头微笑（发松一个ack），李四看到后确认张三在向自己打招呼（李四也进入了establish状态），连接建立。

TCP数据传输

张三喊了一句话（发送data），李四听见后向张三回复自己听见了（发送回一个ack）

如果李四没有及时回复，张三就认为自己的话没被听到，就需要重新喊话（tcp重传），李四就有可能一句话听到了两次，这就是去重，重传和去重工作操作系统的网络内核模块以及处理好了，用户层不用关心。因为TCP是双工的，双方都可以主动发起数据传输，不论是哪方喊话，都需要对方确认才能认为对方收到了自己的喊话。

TCP四次挥手？

张三挥手（发送fin），李四微笑（收到fin后发送ack），此时是半关闭状态，张三（客户端）已经发送了fin请求，不再说话，但是还可以接收李四说的话，李四可以继续说话，等李四不想说话了发送了fin请求时，张三收到并微笑（发送ack），连接断开。

HTTP协议包括哪些请求

  • Get ：对服务器资源的简单请求
  • Post ： 用于发送包含用户提交数据的请求
  • Head ： 类似于GET请求，不过返回的响应中没有具体内容，用于获取报头
  • Delete ：发出一个删除指定文档的请求
  • Trace ： 发送一个请求副本，以跟踪其处理进程
  • Options ：返回所有可用的方法，检查服务器支持哪些方法
  • Connect ：用于ssl隧道的基于代理的请求

简述HTTP中GET和POST的区别

从表面上看：

  • GET请求的数据会附在URL后面，POST的数据放在HTTP包体
  • POST安全性比GET安全性高

从原理性看：

  • 根据HTTP规范，GET用于信息获取，而且应该是安全和幂等的
  • 根据HTTP规范，POST请求表示可能修改服务器上资源的请求

Cookie的作用是什么？它和Session有什么区别？

Cookie和Session都是用来跟踪浏览器用户身份的会话方式，但是两者的应用场景不一样

  • Cookie一般用来保存用户信息
    • 在Cookie中保存已登录过的用户信息，下次访问网站的时候页面可以自动帮你把登录的一些基本信息给填了
    • 一般网站都会有保持登录也就是下次你再访问网站时就不需要重新登录了，这是因为用户登陆的时候我们可以存放一个Token在Cookie中，下次登录根据Token值来查找用户即可（为安全考虑，重新登录一般要将Token重写）
    • 登录一次网站后访问网站其他页面不需要重新登录
  • Session的主要作用是通过服务器记录用户的状态
    • 典型的场景是购物车，当你要添加商品到购物车的时候，系统不知道是哪个用户操作的，HTTP协议是无状态的。服务器给特定的用户创建特点的Session之后就可以标识这个用户并且跟踪这个用户。

Cookie数据保存在客户端（浏览器端），Session数据保存在服务器端，相对来说Session安全性更高。如果要在Cookie中存储一些敏感信息，不要直接写入Cookie中，最好将Cookie信息加密然后使用到的时候再去服务器端解密

HTTP 1.0和HTTP1.1的主要区别是什么？

  • 长连接
    在HTTP/1.0中国，默认使用的是短链接，也就是每次请求都要重新建立一次连接，HTTP是基于TCP/IP协议的，每一次建立或者断开连接都需要三次握手四次挥手的开销，如果每次请求都建立连接，开销比较大。因此最好维持一个长连接，可以用长连接发送多个请求，HTTP1.1开始，默认使用长连接，默认开启Connection：keep-alive。HTTP/1.1的持续连接有非流水线方式和流水线方式。流水线方式是客户在收到HTTP的的响应报文之前就能接着发送新的请求报文。与之相对应的非流水线方式是客户在收到前一个响应后才能发送下一个请求。

  • 错误状态响应码
    在HTTP1.1中新增了24个错误状态响应码，如409(Conflict)表示请求的资源与资源的当前状态发生冲突；410(Gone)状态表示服务器上某个资源被永久性的删除

  • 缓存处理
    在HTTP1.0里主要使用header里的If-Modifed-Since，Expires来作为缓存判断的标准，HTTP1.1则引入了更多的缓存控制策略比如Entity tag，If-Unmodified-Since，If-Match，If-None-Match等更多可供选择的缓存头来控制缓存策略

  • 带宽优化及网络连接的使用
    HTTP1.0中，存在一些浪费带宽的现象，例如客户端只是需要某个对象的一部分，而服务器却将整个对象送过来了，并且不支持断点续传功能，HTTP1.1则在请求头引入了range头域，它允许只请求资源的某个部分，即返回码是206(Partial Content)，这样就方便了开发者自由的选择以便于充分利用带宽和连接。

数据库的三范式是什么

  • 第一范式：强调的是列的原子性，数据表的每一列都不可再分
  • 第二范式：要求实体的属性完全依赖于主关键字。不能存在仅依赖主关键字一部分的属性
  • 第三范式：任何非主属性不依赖于其他非主属性，消除传递依赖

数据库的事务隔离级别

由低到高依次为 Read uncommitted(读未提交）, Read committed（读提交）, Repeatable read（可重复读）, Serializable（序列化）

  • Read uncommitted
    会出现脏读问题，也就是用户读到了尚未提交的事务。举个例子，小明领工资，查看账户发现这个月工资10000，但是财务发现打得钱不对，工资只有8000，于是回滚了事务，修改了金额，再将事务提交，最后小明实际工资只有8000。“小明查工资”这个事务读取了“财务发工资”这个事务还没有提交的数据，即出现脏读。当隔离级别设为read uncommited时，即可能出现此问题。

  • Read committed
    解决了脏读问题，但会出现不可重复读的问题。举个例子，小明拿了8000工资准备去消费，但此时小明女朋友正在网上转账，把小明的8000工资转到另一张卡里，并在小明之前提交了事务，小明拿卡消费时，系统检测到小明卡里没钱了，扣款失败。“小明拿卡消费”这个事务先读取了数据，紧接着“小明女朋友转账”这个事务更新了数据并提交了事务，导致前一个事务再次读取数据时读到了修改后的数据。

    Read committed是大多数数据库的默认级别（SqlServer，Oracle）

  • Repeatable Read
    避免了不可重复读问题，当小明拿卡消费时，一旦读到了数据（8000工资），事务就开始了，小明女朋友就不能对该记录进行修改。
    但此隔离级别可能出现幻读问题。小明女朋友查看小明这个月的信用卡消费记录发现小明花了5000元，而此时小明正消费了1000元并提交了事务，随后小明女朋友打印收据，发现消费金额变成了6000元。
    注：MySQL的默认隔离级别是Repeatable Read

  • Serializable
    Serializable是最高的事务隔离级别，但代价高性能低(完全串行执行)，很少使用，此隔离级别下幻读问题也可以避免。

数据库事务四大特性

  • Atomicity(原子性)：一个事务中的所有操作，要么全部完成，要么都不完成，不会结束在中间某个环节。事务在执行中出现问题，会被回滚到事务开始前的状态，即事务不可分割、不可约简。
  • Consistency(一致性)： 事务开始之前和事务结束以后，数据库的完整性没有被破坏，表示写入的数据必须完全符合所有的预设约束、触发器、级联回滚等。
  • Isolation(隔离性): 数据库允许多个并发事务同时对其数据进行读写和修改的能力，可以防止多个事务并发执行时由于交叉执行而导致数据的不一致，事务隔离的四个级别上面一个问题已经解答。
  • Durability(持久性): 事务处理结束以后，对数据的修改就是永久的，即便系统故障也不会丢失

MySQL索引是怎么实现的？

MySQL一般用B+树作为其索引结构，特点如下：

  • 所有值按顺序存储，每个叶子到根的距离相等
  • 非叶子结点不存储数据，只存储指针索引，叶子结点存储所有数据，不存储指针
  • 每个叶子结点都有指向相邻下一个叶子结点的指针，即顺序访问指针

同样也是使用B+树作为索引结构，叶子结点data域存的是数据记录的地址。数据文件和索引文件分别存储在xxx.MYD和xxx.MYI（xxx表示数据表名），索引文件xxx.MYI保存数据记录的地址，下图为主索引的示意图：

MyISAM中检索索引算法为：首先按照B+树搜索算法搜索，如果找到指定的key，取出其data域的值，再用data域的值为地址去查找对应的数据记录，所以MyISAM的索引方式也称为非聚集索引。

MySQL常用的引擎？

  • InnoDB
    提供了对acid事务的支持，并且提供了行级锁和外键的约束，设计目标是处理大数据容量的数据库系统。MySQL运行的时候，InnoDB会在内存中建立缓冲池，用于缓存数据和索引。但是该引擎不支持全文搜索，同时启动的也比较慢，也不会保存表的行数，当执行select count(*) from table指令时，需要扫描全表。由于锁的粒度小，写操作不会锁定全表，所以在并发度较高的场景下使用会提升效率

  • MyISAM
    MySQL的默认引擎，但不提供对事务的支持，也不支持行级锁和外键。因此当执行插入和更新语句时，即执行写操作时需要锁定这个表，导致效率降低。但与InnoDB不同，MyISAM保存了表的行数，于是进行select count(*) from table语句时，可以直接读取已经保存的值而不需要扫描全表。所以如果表的读操作远远超过写操作且不需要事务支持时，可以将MyISAM作为数据库引擎的首选。

MySQL为何用B+树作为索引的数据结构？

MySQL里索引有两种，Hash和B+树

  • Hash
    Hash算法由于随机性，可能出现哈希冲突，如果执行下面的sql语句：
    Select * from test where name = ‘Leo’
    可以直接对“Leo”按哈希算法算出来一个数组下标，然后直接从数据中取出数据并拿到所对应那一行数据的地址，进而查询那一行数据，但如果执行下面的sql语句
    Select * from test where name > ‘Leo’
    Hash表没有办法做到，因为其特点就是可以快速的精确查询，但是不支持范围查询，如果做成索引，需要全部扫描，速度慢。Hash表更适用于等值查询的场景。
  • B+树
    首先说二叉树，二叉树因为是有序的，所以支持范围查询，但时间复杂度为O(log(N))，为了维持时间复杂度，更新的时间也要是O(log(N))，就需要保持这颗树是完全平衡二叉树，但完全平衡二叉树的维护成本随着数据量的增大而变高，不适用大数据量的查询。
    相比B树，B+树的优势在于B+树的非叶子结点会冗余一份在叶子结点中，并且叶子结点件用指针相连。B树一个结点可以存储多个元素，比完全平衡二叉树的树高低，磁盘IO效率提高了，B+树把非叶子结点进行了冗余，这样提高了范围查找的范围（有一个指针指向下一个结点的叶子结点）

所以B+树可以提高查询索引时的磁盘IO效率，并且可以提高范围查询的效率，并且B+树里的元素也是有序的

索引的最左匹配原则了解吗？

  • 索引可以简单如一个列(a)，也可以复杂如多个列(a,b,c,d)，即联合索引
  • 如果是联合索引，那么key也由多个列组成，同时，索引只能用于查找key是否存在（相等），遇到范围查询就不能进一步匹配了，后续退化为线性查找。
  • 因此，列的排列顺序决定了可命中索引的列数

最左前缀匹配原则就是MySQL会一直向右匹配直到遇到范围查询(>,<,BETWEEN,LIKE)就停止匹配

MySQL如何避免幻读问题？

  • mvcc（多版本并发控制）
    多数数据库都实现了多版本并发控制，并且是通过数据快照实现的，普通的select就是快照读。Session A先执行一个查询，这个查询可以访问任何表，目的是创建一个当前时间点的快照。Session B插入一条数据到数据表read_view中并提交，Session A此时再执行select * from read_view，是看不到SessionB对数据表的任何更改的。因为Session A查询时创建了一个以Select操作的时间为基点的read view，从而避免了幻读的产生。
    原理：将历史数据存一份快照，所以其他事务增加与删除数据，对于当前事务是不可见的。

  • Next-key锁

    Innod行锁分为:

    Next-Key锁包含两部分

    • 记录锁（行锁）
    • 间隙锁

    记录锁时加在索引上的锁，间隙锁是加在索引之间的。
    原理：将当前数据行与上一条数据和下一条数据之间的间隙锁定，保证此范围内读取的数据是一致的。

MySQL有哪些重要的日志文件？

  • 错误日志
    用来记录MySQL服务器运行过程中的错误信息，默认开启无法关闭
    默认情况下，错误日志是存储在数据库的数据文件目录中，名称为hostnae.err，其他hostname为服务器主机名。在MySQL 5.5.7之前，数据库管理员可以删除很长时间之前的错误日志，以节省服务器上的硬盘空间，MySQL 5.5.7之后，服务器将关闭此项功能，只能使用重命名原来的错误日志文件，手动冲洗日志创建一个新的。

  • 查询日志
    查询日志里面记录了数据库执行的所有命令，不管语句是否正确，都会被记录

    • insert 查询为了避免数据冲突，如果此前插入过数据，当前插入的数据如果跟主键或唯一键的数据重复那肯定回报错
    • update 时也会查询因为更新的时候很可能会更新某一块数据
    • delete查询，只删除符合条件的数据

    因此都会产生日志，在并发操作非常多的场景下，查询信息会非常多，那么如果都记录下来会导致IO非常大，影响MySQL性能，所以如果不是在调试环境下，不建议开启查询日志功能

  • redo log(重做日志) 一种缓存机制
    最大程度的避免数据写入时，因为IO瓶颈造成的性能问题
    先将数据写入内存中，再批量把内存中的数据统一刷回磁盘。为了避免将数据刷回磁盘过程中，因为断电或系统故障带来的数据丢失问题，InnoDB采取redo log解决此问题

  • undo log(回滚日志)类似ctrl+z撤销
    用于存储日志被修改之前的值，从而保证如果修改出现异常，可以使用undo log日志来实现回滚操作
    undo log和redo log记录物理日志不一样，它是逻辑日志，可以认为当delete一条记录时，undo log中会记录一条对应的insert记录，反之亦然。当update一条记录时，它记录一条对应相反的update记录，当执行rollback时，就可以从undo log中的逻辑记录读取到相应的内容并进行回滚。undo log默认存放在共享表空间中。

  • bin log(二进制日志)
    是一个二进制文件，主要记录所有数据表结构变更，binlog中记录了对MySQL数据库执行更改的所有操作，并记录了语句发生时间、执行时长、操作数据等其他额外信息，但是它不记录SELECT、SHOW等那些不修改数据的SQL语句。

MySQL主从复制的原理

  • 主库必须开启二进制日志
  • 当有增删改的语句时，会记录到主库的binlog中
  • 主库通过IO线程把binlog里面的内容传给从库的relay binlog（中继日志）（这是MySQL复制是异步复制的原因）
  • 从库的sql线程负责读取它的relay log里的信息并应用到数据库中

Java中的反射

Posted on 2020-03-13 In Java

反射的概念

Java中的反射首先是能获得Java中要反射类的字节码，获得字节码有三种方式：

  • Class.forName(className)
    • 例子：Class clazz = Class.forName(“类的全路径”);（最常用）
  • 类名.class
    • 例子：Class clazz = Person.class;
  • this.getClass()
    • 例子：Person p = new Person(); Class clazz = p.getClass();

然后将字节码中的方法，变量，构造函数等映射成相应的Method、Filed、Constructor等类，这些类提供了丰富的方法可以被我们使用.

创建对象的两种方法

Class对象的newInstance()

使用Class对象的newInstance()方法来创建该Class对象对应类的实例，该Class对象对应的类有默认的空构造器

调用Constructor对象的newInstance()

先使用 Class 对象获取指定的 Constructor 对象，再调用 Constructor 对象的 newInstance()方法来创建 Class 对象对应类的实例,通过这种方法可以选定构造方法创建实例。

动静态代理的区别，什么场景使用

静态代理通常只代理一个类，动态代理是代理一个接口下的多个实现类。静态代理事先知道要代理的是什么，而动态代理不知道要代理什么东西，只有在运行时才知道。动态代理是实现JDK里的InvocationHandler接口的invoke方法，但注意的是代理的是接口，也就是你的业务类必须要实现接口，通过Proxy里的newProxyInstance得到代理对象。

还有一种动态代理cglib，代理的是类，不需要业务类继承接口，通过派生的子类来实现代理。通过在运行时，动态修改字节码达到修改类的目的。

AOP就是基于动态代理实现的，Spring框架、Hibernate框架等都是动态代理的使用例子。

    状态就是当前的索引值，选择就是加这个数还是不加，这个数本身和前一个数加这个数比大小来获得当前索引能取得的子序和最大值（有点像打家劫舍第一题），状态转移方程如下：

    dp[i] = Math.max(dp[i - 1] + nums[i], nums[i])

Leetcode 1013 Partition Array Into Three Parts With Equal Sum(将数组分成和相等的三个部分)

Posted on 2020-03-11 In leetcode

Difficulty : Easy

Type: Array

Language : Java

Clarification

Given an array A of integers, return true if and only if we can partition the array into three non-empty parts with equal sums.

Formally, we can partition the array if we can find indexes i+1 < j with (A[0] + A[1] + ... + A[i] == A[i+1] + A[i+2] + ... + A[j-1] == A[j] + A[j-1] + ... + A[A.length - 1])

Constraints:

  • 3 <= A.length <= 50000
  • -10^4 <= A[i] <= 10^4

Possible Solutions

  • 双指针

    三个和相等的非空部分，三部分总和一定可以整除3，所以先计算数组的总和能不能整除3，如果不能返回false。如果可以，用双指针遍历数组，左指针初始值为0，右指针初始值为数组长度 - 1，循环条件为left + 1 < right(防止数组分成两个部分)，只要左指针的和不为sum/3，左指针就遍历下一个位置并加上当前位置的值到总和里，右指针同理只不过是遍历前一个位置，如果左右指针都等于sum/3，中间也一定等于sum/3，此时return true，其他情况return false

    Left + 1 < Right的约束就是要中间有剩下的元素，以免数组划分成两部分，中间没有元素。
